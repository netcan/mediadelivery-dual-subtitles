## Context

当前宿主页悬浮字幕时间轴已经具备查看、定位当前、点击跳转、拖动和尺寸记忆等能力，但它仍然是一个偏“浏览/控制”工具，而非“内容摘录”工具。要支持分享，核心问题不在于字幕数据获取，而在于如何在不破坏现有跳转交互的前提下，引入多选状态、批量操作和稳定的图片生成链路。

现有时间轴条目由 `host.js` 统一渲染，这意味着最自然的实现位置也是宿主页层：在这里维护多选状态，并根据选中的时间轴条目数据直接生成图片，而不是反向依赖 iframe 内 DOM 或页面截图。

## Goals / Non-Goals

**Goals:**
- 在宿主页悬浮字幕时间轴中支持多选字幕条目。
- 支持 `Ctrl/Cmd` 单条切换与 `Shift` 连续选择。
- 支持根据选中的字幕条目生成结构化双语分享图片。
- 保持现有单击跳转、定位当前和暂停自由滚动体验。

**Non-Goals:**
- 不实现服务端分享链接或上传能力。
- 不首期支持复杂模板市场、主题编辑器或富样式自定义。
- 不把选择状态同步回 iframe 内时间轴。

## Decisions

### 1. 选择状态只维护在宿主页悬浮时间轴
- 决策：仅在 `host.js` 中维护 `selectedIndexes` 与 `selectionAnchorIndex`。
- 原因：分享功能只在宿主页悬浮窗暴露，避免 iframe 与宿主页之间增加复杂双向选择同步。
- 备选方案：让 iframe 内外共用选择状态。缺点是会显著增加消息同步和状态一致性成本。

### 2. 普通点击仍保留“跳转”，多选使用修饰键显式触发
- 决策：普通点击保持当前“跳转到该字幕时间点”；`Ctrl/Cmd` 点击用于增删单条；`Shift` 点击用于从 anchor 到当前条连续选择。
- 原因：这样不破坏现有主交互路径，同时为高级选择保留清晰手势。
- 备选方案：点击默认进入选择模式。缺点是会削弱时间轴作为导航工具的直觉性。

### 3. 分享图片通过 Canvas 本地绘制生成
- 决策：根据选中的字幕数据在前端 `canvas` 中重新排版并导出 PNG，而不是截图 DOM。
- 原因：本地绘制更稳定，不受页面样式、滚动位置、跨 iframe 或浏览器截图权限影响。
- 备选方案：DOM 截图。缺点是样式依赖强、跨层截图不稳、结果不可控。

### 4. 首期先支持下载 PNG，后续再扩展复制/系统分享
- 决策：首期分享出口最小化为“下载图片”。
- 原因：浏览器兼容性最好，验证成本最低；后续可以在同一渲染链路上扩展到 Clipboard 或 Web Share。
- 备选方案：一开始同时做下载、复制和系统分享。缺点是兼容性处理更多，首期复杂度偏高。

## Risks / Trade-offs

- [多选与跳转手势冲突] → 保持普通点击仍为跳转，多选只在修饰键下生效，并提供清空选择按钮。
- [长字幕导致图片过高] → 首期限制导出条目数或画布高度，并对文本做自动换行。
- [中英混排排版不稳定] → 使用固定版式和统一字体栈，在 Canvas 中自行计算文本块高度。
- [用户误以为截图的是当前窗口样式] → 按钮文案明确为“分享图片”，并让导出结果保持稳定模板风格。
