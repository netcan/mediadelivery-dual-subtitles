## Context

当前实现里，Python Provider 会先把所有字幕段依次送入 TTS，待全部分段生成完成后再统一混音并写出单个 `.wav` 文件；插件侧也假设任务结果里只会存在一个完整 `audioUrl`，因此“启用中文配音”开关只有在任务完全结束后才会解锁。这个模型实现简单，但首播等待时间与视频时长线性相关，对长视频体验较差。

本次变更需要在不引入 ASR、不改动“现有中文字幕驱动配音”的前提下，把整条等待升级为“部分分段就绪即可播”。它是一个跨 Provider 输出协议、任务状态模型、插件轮询逻辑和音频播放编排的联动改造，适合先通过设计明确边界与取舍。

## Goals / Non-Goals

**Goals:**
- 降低中文配音首播等待时间，让前几段生成完成后即可开始播放。
- 保持当前轮询式任务契约，不引入真正的字节流推送协议。
- 让 Provider 在运行中暴露分段级结果，同时保留最终完整结果。
- 让插件在分段未全部就绪时，仍能稳定处理暂停、继续、跳转和回退。

**Non-Goals:**
- 不在本次变更中引入 ASR、说话人分离或新的字幕来源。
- 不直接升级为 HLS、MSE 或其他复杂流媒体协议。
- 不追求首版就覆盖所有极端 seek/倍速场景下的最优体验，只要求行为明确、可恢复。

## Decisions

### 决策 1：采用“分段文件 + 任务轮询增量结果”而不做真流式字节输出
Provider 继续使用现有 `POST /jobs` + `GET /jobs/:id` 模型，但在任务执行过程中持续更新 job 结果，暴露已完成的分段文件地址、连续可播放范围和可提前播放状态。

**原因：**
- 与当前实现最接近，不需要重做传输协议。
- 便于调试、失败重试与结果缓存。
- 比起“不断增长的单个 wav 文件”更符合浏览器和静态文件服务的能力边界。

**替代方案：**
- 真流式字节输出：浏览器消费和 Range/缓存行为复杂，首版风险高。
- HLS/MSE：媒体能力更强，但对当前插件与 Provider 的改造跨度过大。

### 决策 2：Provider 同时维护“运行态分段结果”和“完成态整段结果”
任务运行中输出分段文件；任务完成后可选地继续整理为单个最终音频文件，并在最终结果中返回 `audioUrl`。

**原因：**
- 运行态满足边生成边播。
- 完成态便于复播、持久化和保留简单播放模式。
- 可降低插件在“首次播放”和“再次播放”时的逻辑复杂度。

### 决策 3：插件以“分段队列 + 预取 + 可播放窗口”驱动播放
插件不再仅依赖单个 `audioUrl`，而是维护当前播放分段、下一个待预取分段及连续可播放窗口。仅当从时间轴起点或当前目标时间开始已有最小可播放窗口时，才允许用户启用中文配音。

**原因：**
- 可在首版避免“刚勾上就立刻卡住”的体验。
- 允许后续继续优化预取与缓冲策略。

**替代方案：**
- 只要第一段完成就允许播放：实现最简单，但容易在数秒后立刻断流。

### 决策 4：未就绪分段采用“回退原声或等待”策略，而不是强行静音等待
当用户 seek 到尚未生成完成的位置，插件应暂停外挂配音，并明确回退到原声或显示等待状态；待目标分段可用后再恢复外挂配音。

**原因：**
- 行为清晰，能减少“无声卡住”的困惑。
- 与现有原视频音频恢复逻辑兼容。

## Risks / Trade-offs

- [播放编排复杂度提升] → 先限定为顺序分段播放 + 基础预取，完成态仍保留整段结果作为回退路径。
- [任务状态结构变复杂] → 维持现有字段兼容，新增分段与可播放元数据字段，避免破坏旧 Provider 调用。
- [seek 到未生成区域的体验不稳定] → 明确规定回退原声或等待，而不是要求首版实现无缝跳播。
- [分段文件数量变多] → 以任务目录或规范命名管理，并在后续考虑清理策略。
- [日志与调试信息增多] → 在 Provider 中增加 job/segment 级状态输出，避免只看到底层模型进度条。

## Migration Plan

1. 扩展 Provider 的 job/result 模型，允许持久化分段状态、连续可播区间和完成态结果。
2. 调整 Python Provider 生成流程，先逐段落盘并更新 job，再在完成时可选生成整段结果。
3. 扩展插件端任务轮询与状态归一化逻辑，识别“部分可播”状态与分段元数据。
4. 重构插件外挂音频播放逻辑，从单资源播放改为分段队列播放，并保留完成态单资源兼容路径。
5. 增加长视频与 seek 场景的验证用例，确认在未就绪区间的回退行为明确可预期。

## Open Questions

- 最小可播放窗口应该基于“首 N 秒”还是“首 N 个分段”？
- 最终整段结果是强制生成还是允许按配置关闭，以减少等待和磁盘占用？
- 分段文件与最终整段文件的清理策略是否要纳入本次变更，还是后续单独处理？
