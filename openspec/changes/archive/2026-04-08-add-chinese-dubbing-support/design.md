## Context

当前扩展仅覆盖 MediaDelivery iframe 内的字幕读取与双语字幕叠加，不负责音轨管理，也没有与外部模型服务交互的抽象。用户的新目标是在“英文原声 + 中文字幕”已存在的前提下，为视频生成并播放中文配音，同时允许接入云端 API、自定义 HTTP API 或本地 `localhost` 模型服务。

本次变更的核心约束如下：

- 首期只依赖现有中文字幕驱动配音生成，不实现 ASR。
- 当前仓库是 Chrome 扩展，适合承担播放器接管、设置管理和任务调用，不适合在扩展内直接运行重型 ASR/TTS 模型。
- MediaDelivery 嵌入页可能没有原生多音轨切换入口，因此需要支持“外挂音频 + 原视频静音/压低”的方案。
- 后续需要为 ASR、说话人分离和更细粒度对齐预留接口，但首期不引入这些复杂度。

相关干系对象包括：使用扩展的最终用户、接入外部模型服务的部署者，以及未来可能维护本地模型服务适配器的开发者。

## Goals / Non-Goals

**Goals:**

- 允许用户在扩展中配置中文配音任务所需的模型服务提供方。
- 基于现有中文字幕发起中文配音生成任务，并查询任务状态与结果。
- 在 MediaDelivery 播放器中稳定切换到生成后的中文配音，并保持中文字幕显示。
- 为未来引入 ASR 增强、更多 Provider 类型和更复杂配音元数据保留接口扩展空间。

**Non-Goals:**

- 不在本次设计中实现 ASR、说话人分离、口型对齐或背景音乐分离。
- 不在扩展内直接运行本地大模型或语音模型推理。
- 不定义具体某一家第三方模型提供商的私有 API 协议，只定义扩展侧需要的最小通用契约。
- 不处理站点已有多音轨时的深度播放器内核集成；首期以外挂中文音频同步播放为主。

## Decisions

### 决策 1：采用“扩展控制层 + 外部任务服务”架构

扩展负责读取视频与字幕、收集 Provider 配置、发起任务、轮询状态并接管播放；外部服务负责配音流水线的生成。

**原因：**

- MV3 service worker 生命周期不稳定，不适合长时推理任务。
- 语音模型资源占用大，在浏览器内执行会显著影响播放体验。
- 外部服务可以更灵活地接入云端 API、本地模型和自定义协议。

**备选方案：**

- 在扩展内直接调用 TTS API 并实时合成：实现简单，但长视频、失败恢复和结果缓存都较差。
- 在扩展内运行本地模型：可离线，但工程复杂度与浏览器资源压力过高，不适合作为首期方案。

### 决策 2：首期以“中文字幕驱动配音”作为输入基线

任务输入以现有中文字幕及其时间轴为主，生成适合口播的中文配音稿和对应中文配音音频。

**原因：**

- 用户当前输入条件明确为“英文原音 + 中文字幕”，无需先做 ASR 才能形成闭环。
- 能更快验证产品价值，并降低首期模型调用成本与接口复杂度。
- 仍可在任务协议中预留 `timingSource`、`asrResult` 等字段，后续再增强。

**备选方案：**

- 首期直接加入 ASR：自然度和对齐潜力更好，但实现跨度大、风险高，不符合当前阶段。

### 决策 3：Provider 配置采用统一抽象，兼容云端、自定义和本地服务

扩展统一存储 Provider 类型、`baseURL`、`apiKey`、翻译模型、TTS 模型和可选的音色配置，由任务请求按统一字段提交。

**原因：**

- 避免实现绑定单一厂商。
- 兼容自建服务、OpenAI-compatible 接口和 `localhost` 模型服务。
- 后续接入更多生成参数时，只需扩展配置结构和请求载荷。

**备选方案：**

- 为每个提供商单独做一套 UI 与请求格式：首期上线快，但长期维护成本高。

### 决策 4：播放侧默认采用“原视频静音/压低 + 外挂中文音频同步”

扩展在启用中文配音后，创建独立音频对象承载中文配音流，并将其与视频的播放、暂停、跳转、倍速保持同步。

**原因：**

- 当前项目没有可靠的原生音轨切换能力。
- 外挂音频方案不依赖底层播放器是否暴露多音轨 API。
- 能最大化复用现有字幕叠加逻辑。

**备选方案：**

- 直接替换原视频音轨：需要更深入播放器或流协议接管，复杂度高。
- 仅播放中文配音，不与视频同步控制绑定：用户体验不可接受。

### 决策 5：结果契约至少包含音频地址、字幕地址和分段元数据

任务完成后，扩展至少需要拿到中文配音音频地址、可选的输出字幕地址以及用于同步和排障的分段元数据。

**原因：**

- 音频地址用于播放接入。
- 字幕地址可用于后续切换为配音稿字幕或保留原中文字幕。
- 分段元数据可支持未来做漂移校正、局部重试和 ASR 增强。

**备选方案：**

- 仅返回一个音频地址：短期可行，但不利于后续调试与扩展。

## Risks / Trade-offs

- [外挂音频长时间播放可能产生漂移] → 在同步控制中加入定期校时机制，并保留结果元数据中的偏移信息。
- [用户配置的自定义 API 质量参差不齐] → 在 Provider 配置和任务状态中明确错误提示，区分网络错误、认证错误和模型错误。
- [中文字幕并非口播友好文本] → 任务契约中明确“先生成可配音稿再生成音频”，避免直接将字幕逐字念出。
- [本地 `localhost` 服务涉及权限与可达性问题] → 在权限策略与 UI 中提示可访问域名限制，并将连接测试作为配置流程的一部分。
- [不同站点/播放器结构差异影响接入稳定性] → 继续限定首期目标为 `https://iframe.mediadelivery.net/embed/*`，不扩展到更广泛站点。

## Migration Plan

1. 在扩展设置中引入 Provider 配置存储，同时保持现有字幕设置兼容。
2. 扩展控制面板新增中文配音任务入口和播放切换入口，但默认不影响现有双语字幕行为。
3. 增加结果接入和外挂音频同步逻辑后，以功能开关方式逐步验证。
4. 若任务接口或同步逻辑不稳定，可通过隐藏中文配音入口回滚到纯字幕模式，无需迁移现有字幕数据。

## Open Questions

- 是否需要在首期就支持“保留原中文字幕”与“切换到配音稿字幕”两种显示模式？
- Provider 配置是否允许多个预设并在面板中快速切换？
- 自定义 API 的最小契约是否只覆盖异步任务模式，还是要兼容同步返回结果？
- 原声在启用中文配音时默认应完全静音，还是提供压低音量的混合模式？
