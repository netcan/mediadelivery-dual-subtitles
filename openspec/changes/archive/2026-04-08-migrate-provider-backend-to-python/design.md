## Context

当前仓库原先有一个 Node.js 版 `local-provider`，它完成了扩展契约、任务轮询、结果文件输出和基础本地 TTS 适配。但随着项目开始接入 VoxCPM 这类原生提供 Python 推理接口的模型，Node 版后端需要额外依赖 HTTP 网关或子进程桥接，导致部署路径变长、调试链路更绕，也不利于直接复用 Python 生态中的模型加载、权重管理和推理参数。

本次变更的目标不是修改扩展协议，而是在保持扩展侧 `POST /jobs`、`GET /jobs/:id` 契约稳定的前提下，把本地 Provider 的主推荐后端迁移到 Python，并让 VoxCPM 成为首期唯一的真实模型路径。同时，模型、推理参数和默认运行配置由 Provider 内部统一管理，插件前端仅保留少量与使用体验直接相关的选择项，其中首期保留 Provider 地址配置与音色选择。涉及角色包括：扩展使用者、本地模型部署者、后续维护 Python Provider 的开发者。

## Goals / Non-Goals

**Goals:**
- 提供 Python 版本地 Provider 服务，并保持与扩展现有 HTTP 契约兼容。
- 直接集成 VoxCPM Python SDK，避免必须先搭一层独立模型网关。
- 将模型与推理配置下沉到 Provider 内部，简化插件前端配置面。
- 保留插件侧 Provider 地址配置，允许用户连接本机或局域网内指定 Provider。
- 提供 Provider 能力查询，让前端可读取可选音色并展示给用户选择。
- 复用已有的字幕驱动配音思路，包括分段预处理、输出音频文件、可选字幕文件和最小元数据。
- 完成从 Node Provider 到 Python Provider 的收敛，降低后续维护复杂度。

**Non-Goals:**
- 不修改扩展的 Provider 配置 UI、任务请求体或播放同步逻辑。
- 不在本次变更中实现 ASR、多人说话人分离、口型驱动或复杂混音。
- 不要求首期就支持多模型切换或其他 Python TTS 模型。
- 不在本次变更中继续维护 Node 版 Provider 的兼容实现。

## Decisions

### 决策 1：保留扩展契约不变，仅替换本地 Provider 的推荐实现
Python Provider 继续暴露 `POST /jobs`、`GET /jobs/:id`、`GET /files/:name`、`GET /health`，保持扩展侧零改动即可接入。

**原因：**
- 扩展端已经完成契约和播放器联调，继续变更会扩大影响面。
- 用户的核心诉求是更容易维护与集成模型，而不是重写扩展交互逻辑。

**备选方案：**
- 让扩展直接调用 Python 模型：会把模型依赖和跨域问题带回浏览器侧，不合适。

### 决策 2：Python Provider 采用“服务层 + VoxCPM 适配层 + 输出层”结构
服务层负责 HTTP 路由、任务管理与能力查询；适配层负责 VoxCPM 调用；输出层负责音频/字幕文件写入和静态暴露。

**原因：**
- 这与既有 Provider 职责划分一致，迁移成本更低。
- 可以把 VoxCPM 逻辑从服务路由中隔离出来，便于测试和后续维护。

**备选方案：**
- 单文件脚本直接处理所有逻辑：首期快，但后续接其他模型时会很难维护。

### 决策 3：Python Provider 首期仅支持 VoxCPM
首期 Python Provider 直接调用 `VoxCPM.from_pretrained(...).generate(...)` 完成配音，不再要求兼容 CosyVoice 等其他模型。

**原因：**
- VoxCPM 提供原生 Python API，集成路径更短。
- 与“Provider 后端用 Python 更容易维护、更容易接模型”的目标一致。
- 收敛到单模型路径有利于先把真实可用性、部署文档和联调稳定性做扎实。

**备选方案：**
- 首期并行支持多个模型：灵活，但会扩大实现和验证范围。

### 决策 4：模型配置下沉到 Provider，前端保留 Provider 地址与音色选择
插件前端不再直接维护 `ttsModel`、推理参数或模型路径等细节；Provider 通过配置文件或环境变量管理这些设置，并通过能力接口暴露可选音色列表给插件选择。前端仍然允许用户配置 Provider 的 IP / 端口 / Base URL，以便连接本机或远端部署的 Provider。

**原因：**
- 模型参数属于部署细节，放在前端会增加误配概率。
- Provider 地址属于连接配置，保留在前端更符合用户心智，也便于切换不同机器上的 Provider。
- 音色是用户体验相关配置，保留给前端选择更合理。
- Provider 暴露能力接口后，前端可动态渲染可选音色，而不必硬编码。

**备选方案：**
- 前端继续暴露完整模型表单：灵活，但复杂度和维护成本更高。

### 决策 5：保留异步任务模式，允许同步作为补充
即使 Python 内部直接调用模型，也继续支持任务创建后轮询完成结果；对于短任务可以允许同步返回。

**原因：**
- 模型推理时间受机器与显存影响较大，异步更稳。
- 扩展现有轮询逻辑已经成熟，继续复用即可。

**备选方案：**
- 仅支持同步：实现简单，但容易阻塞请求并影响长字幕任务稳定性。

### 决策 6：字幕驱动的预处理继续作为 Provider 内部标准步骤
无论底层模型是 VoxCPM 还是后续其他 Python 模型，都在合成前执行字幕清洗、合并与基础口语化预处理。

**原因：**
- 这是提升自然度的关键步骤，不应绑死在某一个模型实现里。
- 模型替换时能复用同一输入整形策略。

**备选方案：**
- 完全依赖模型自己修正文案：实现更省事，但配音质量和一致性不可控。

## Risks / Trade-offs

- [Python 依赖更重] → 提供明确的虚拟环境、依赖安装和模型准备文档，优先先支持一条 VoxCPM 推荐路径。
- [模型首帧加载慢] → 允许 Provider 在进程启动时预加载模型，避免每个任务重复初始化。
- [迁移过程中残留旧 Node Provider 引用] → 同步删除旧目录、脚本和文档，避免双主线并行演进。
- [前端与 Provider 的音色列表不一致] → 通过 `GET /capabilities` 或等价接口由 Provider 动态下发可选音色。
- [本地机器资源不足导致任务失败] → 保留标准化失败状态和健康检查接口，帮助快速定位问题。

## Migration Plan

1. 在仓库中新增 Python Provider 目录、启动入口与依赖清单。
2. 复刻现有扩展契约到 Python 服务层，并补充能力查询接口与静态文件输出。
3. 实现 VoxCPM 适配器与 Provider 配置管理，完成端到端联调。
4. 更新文档，将 Python Provider 作为默认推荐部署方式，并说明前端配置 Provider 地址并选择音色。
5. 删除旧的 Node 版 `local-provider`，统一由 Python Provider 承担本地配音主路径。

## Open Questions

- Python Provider 首期选用标准库 HTTP、FastAPI 还是 Flask 更合适？
- VoxCPM 模型是否需要在服务启动时常驻预热，还是允许按首次请求懒加载？
- 音色列表是完全静态配置，还是允许从模型运行时动态探测？
