## Context

当前插件已经具备三块与该功能直接相关的基础能力：一是右上角扩展面板可承载新的信息区块；二是 `getTrackCues()` 已能把原生字幕轨、导入字幕和解析后的轨道统一成可消费的 cue 列表；三是视频跳转后，现有 `seeked` / `timeupdate` / 中文配音同步逻辑已经能处理字幕与配音的后续刷新。因此，字幕时间轴功能的核心不是播放器控制，而是“如何组织和展示双语 cue 列表”。

首期目标是做一个实用的字幕导航器：列表按主字幕时间轴展示时间戳、主字幕和副字幕，点击后跳转到该条字幕起点，并高亮当前播放中的条目。为了控制复杂度，v1 使用当前主字幕轨作为时间轴锚点，再为每条主字幕匹配时间上最接近或重叠的副字幕文本。

## Goals / Non-Goals

**Goals:**
- 在现有面板中增加字幕时间轴列表。
- 允许用户点击时间轴条目跳转到对应视频时间点。
- 以主字幕轨为时间轴基准，并展示对应副字幕文本。
- 高亮当前播放位置对应的时间轴条目。
- 复用现有跳转后的字幕与中文配音同步链路。

**Non-Goals:**
- 不实现全文搜索、筛选、批量展开折叠或虚拟滚动优化。
- 不分别为主/副字幕各自维护独立时间轴。
- 不在本次引入字幕编辑、导出或时间轴校正工具。
- 不尝试完美重建双轨 union cue 时间轴。

## Decisions

### 1. 使用主字幕轨作为时间轴锚点
- 决策：时间轴条目以当前主字幕轨的 cue 列表为准，再附加匹配到的副字幕文本。
- 原因：实现简单、可预测，且与用户当前正在看的主字幕视角一致。
- 备选方案：用双轨 union cue 重建完整时间轴；更完整，但复杂度明显更高。

### 2. 时间轴放在现有面板中，而不是覆盖在视频上
- 决策：将时间轴列表放入右上角现有面板中，作为一个新增区块。
- 原因：避免遮挡视频，也能复用现有开关与配置入口。
- 备选方案：在视频上额外做侧边栏/弹层；更显眼，但更容易干扰播放画面。

### 3. 点击条目只设置 `video.currentTime`
- 决策：点击时间轴后直接设置 `video.currentTime = cue.start`，不单独操控外挂音频对象。
- 原因：现有 `seeked` 和配音同步链路已经能处理跳转后的状态修正。
- 备选方案：同时手动同步外挂音频；会引入重复状态更新。

### 4. 当前项高亮由播放时间驱动
- 决策：根据当前 `video.currentTime` 计算激活 cue，并在时间轴中高亮对应条目。
- 原因：和现有字幕渲染一样，当前播放时间本来就是唯一可信来源。
- 备选方案：点击后单独维护选中状态；在播放继续推进后容易失真。

## Risks / Trade-offs

- [主/副字幕时间轴不完全对齐] → 首期明确以主字幕为锚点，并用重叠/相近时间匹配副字幕，接受轻微错位。
- [列表过长影响面板性能] → v1 先做完整列表，后续如有需要再加虚拟滚动或折叠优化。
- [点击跳转时中文配音短暂重同步] → 复用现有 `seeked` 与同步逻辑，不增加独立配音控制分支。
- [当前项自动滚动过于打扰] → 首期可以只做高亮，自动滚动保持温和或后续再优化。

## Migration Plan

1. 在面板中增加字幕时间轴区块与基础样式。
2. 基于主字幕轨构建时间轴项，并匹配副字幕文本。
3. 为时间轴条目接入点击跳转与当前项高亮。
4. 验证原生字幕、导入字幕、中文配音启用和长列表场景。
5. 如需回滚，只需移除面板区块与相关列表状态，不涉及外部协议或数据迁移。

## Open Questions
