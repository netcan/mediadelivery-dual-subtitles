## Context

当前扩展已经能够配置 Provider、发起中文配音任务并消费标准化结果，但仓库中还没有真实可运行的 Provider 服务。用户希望优先使用本地部署的 TTS 模型，而不是先接云端 API，因此需要在现有扩展接口之上补一层本地服务，用于接收字幕驱动的配音任务、调用本地 TTS 模型并输出音频结果。

这次设计的关键约束如下：

- 扩展侧接口已经固定为 `POST /jobs` 与 `GET /jobs/:id`，本地服务应尽量兼容现有契约，避免回头修改扩展。
- 首期优先本地 TTS，不实现 ASR、说话人分离、复杂混音或背景音保留。
- 本地服务需要能够适配不同 TTS 模型，首期优先以 CosyVoice 风格的本地部署方式为目标。
- 模型推理不应在扩展内执行，应由本地服务在 `localhost` 处理。

涉及对象包括：扩展使用者、本地模型部署者，以及未来可能新增其他本地 TTS 适配器的开发者。

## Goals / Non-Goals

**Goals:**

- 提供一个本地可运行的 Provider 服务，与当前扩展的中文配音任务接口对齐。
- 为本地 TTS 模型建立统一适配层，首期优先支持 CosyVoice 风格模型接入。
- 生成并暴露本地可访问的中文配音结果，包括音频文件、可选字幕文件和最小元数据。
- 让用户仅通过在扩展中配置 `localhost` 地址与模型名，即可联调中文配音流程。

**Non-Goals:**

- 不在本次设计中实现 ASR、说话人分离、背景音提取或口型对齐。
- 不要求首期支持多个模型同时热切换或高级调度队列。
- 不把所有模型推理逻辑都硬编码到扩展中。
- 不承诺首期实现完整的多人角色配音，仅聚焦单轨中文配音结果。

## Decisions

### 决策 1：新增独立本地 Provider 服务，而不是让扩展直接调用模型

本地 TTS 模型通过独立服务运行，扩展只与该服务的 HTTP 接口交互。

**原因：**

- 本地模型推理资源占用高，不适合在 MV3 扩展内运行。
- 独立服务更容易做文件输出、结果缓存、错误处理和模型替换。
- 保持扩展与模型运行时解耦，后续接入其他本地或云端 Provider 更容易。

**备选方案：**

- 扩展直接调用 Python/本地命令：浏览器权限、稳定性和维护成本都较差。

### 决策 2：首期采用“统一任务接口 + 同步/异步兼容”返回模式

服务对外保持 `POST /jobs` 与 `GET /jobs/:id`，允许模型较快时直接返回结果，也允许先创建任务再轮询。

**原因：**

- 与当前扩展的既有设计一致，减少前端修改。
- 本地模型在不同机器上的推理时长差异较大，兼容同步与异步更稳。

**备选方案：**

- 只支持同步：实现简单，但长视频或大模型很容易阻塞。
- 只支持异步：更通用，但首期开发与调试成本更高。

### 决策 3：建立可替换的本地模型适配层

服务内部定义统一的 TTS 适配器接口，首期实现 CosyVoice 风格适配器，并为 Fish Speech、MeloTTS 等模型预留扩展位。

**原因：**

- 用户已经明确优先考虑本地部署 TTS 模型，不希望被单一实现绑死。
- 不同模型的输入输出格式不同，适配层有利于隔离差异。
- 后续替换或新增模型时无需重写任务 API。

**备选方案：**

- 直接把某个模型命令写死在任务处理流程里：首期快，但难维护、难扩展。

### 决策 4：结果以本地静态文件 URL 形式暴露

服务生成的音频和可选字幕文件写入本地输出目录，并通过本地静态路由或文件服务方式提供给扩展。

**原因：**

- 扩展当前消费的是 `audioUrl` / `subtitleUrl`，用 URL 能直接接入现有逻辑。
- 便于调试、复用和结果缓存。

**备选方案：**

- 在接口内直接返回大体积音频二进制：对扩展和任务轮询都不友好。

### 决策 5：字幕驱动的口语化改写作为服务内部预处理步骤

本地 Provider 在调用 TTS 前，先对中文字幕分段做清洗与口语化改写接口预留；首期允许先以较轻的文本处理策略打底。

**原因：**

- 直接朗读字幕往往不够自然。
- 即使首期不接强文本模型，也应预留“配音稿生成”阶段。

**备选方案：**

- 完全跳过改写：实现快，但配音自然度较差。

## Risks / Trade-offs

- [本地模型部署依赖复杂] → 提供清晰的模型准备与服务启动文档，并优先支持单一推荐路径。
- [不同机器推理性能差异大] → 兼容异步任务模式，避免阻塞扩展交互。
- [本地文件暴露路径管理不当] → 统一输出目录并限制仅暴露必要结果文件。
- [直接朗读字幕会显得机械] → 预留配音稿改写步骤，后续可接更强文本模型。
- [首期仅支持单轨中文配音，效果有限] → 在文档中明确首期限制，为说话人分离和高级混音保留后续扩展位。

## Migration Plan

1. 在仓库中新增本地 Provider 服务及其运行说明。
2. 让扩展默认支持将 `localhost` 作为 Provider Base URL。
3. 完成基础联调后，通过文档指导用户先本地部署推荐模型与服务。
4. 若本地 Provider 不稳定，可临时关闭扩展中的中文配音入口，保留纯字幕能力。

## Open Questions

- 首期本地 Provider 服务选用 Node.js 还是 Python 作为主实现语言更合适？
- CosyVoice 适配器是否直接调用本地 Python 进程，还是通过单独模型网关接入？
- 是否需要在首期就支持任务结果持久化清理策略？
- 本地字幕改写是使用简单规则还是引入本地文本模型作为可选增强？
