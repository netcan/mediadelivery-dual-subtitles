## Context

当前插件已经管理视频元素、字幕覆盖层、全屏挂载位置以及中文配音同步逻辑，但还没有统一的键盘播放控制入口。由于目标站点播放器处于网页/iframe 场景中，插件更适合在拥有目标视频元素的内容脚本上下文里监听键盘事件，并复用现有 `video` 播放、暂停、跳转后的同步链路，而不是额外维护一套独立音频状态机。

本次范围限定为“全屏播放模式下”的快捷键支持：仅当插件当前管理的视频处于全屏上下文时，空格键切换播放/暂停，左右方向键执行 ±5 秒跳转；普通页面浏览、表单输入和非全屏场景不纳入拦截范围。

## Goals / Non-Goals

**Goals:**
- 为插件当前管理的视频增加全屏模式下的键盘快捷键支持。
- 让空格键通过统一入口切换播放/暂停。
- 让左右方向键在全屏模式下执行固定 5 秒跳转，并复用现有字幕和中文配音同步逻辑。
- 避免拦截输入框、文本域和可编辑区域中的常规键盘输入。

**Non-Goals:**
- 不修改非全屏状态下的键盘行为。
- 不引入可配置的快捷键映射、可变跳转步长或更多播放器热键。
- 不接管站点原生播放器的全部键盘系统，只补齐插件自身需要保证的最小行为。

## Decisions

### 1. 在内容脚本中注册单一 `keydown` 监听
- 决策：在管理目标视频的内容脚本上下文注册单一键盘监听，统一处理空格、左方向键和右方向键。
- 原因：当前插件的播放、字幕和配音同步都围绕 `state.video` 展开，直接在同一上下文处理按键可最小化状态分散。
- 备选方案：
  - 依赖站点原生播放器快捷键：不同站点/iframe 行为不一致，插件无法保证一致体验。
  - 分别对 overlay 或自定义面板绑定事件：焦点易丢失，无法覆盖纯视频全屏观看场景。

### 2. 仅在“当前全屏元素包含受管视频”时生效
- 决策：通过已有全屏检测能力判断当前全屏元素是否包含 `state.video`，只有满足条件才处理按键。
- 原因：这样可以将快捷键严格限制在当前插件控制的视频上下文，避免影响同页其他内容或普通滚动浏览。
- 备选方案：全局始终监听并生效；该方式冲突面过大，容易误伤页面交互。

### 3. 直接驱动 `video`，复用现有同步事件
- 决策：空格键直接调用 `video.play()` / `video.pause()`，左右方向键直接更新 `video.currentTime`，不单独控制外挂音频对象。
- 原因：现有逻辑已经在 `play`、`pause`、`seeked`、`timeupdate` 等事件上同步字幕与中文配音，直接驱动视频即可复用已有链路。
- 备选方案：同时直接控制 `dubbedAudio`；该方式容易造成双重状态更新和不同步。

### 4. 显式跳过可编辑输入目标
- 决策：当事件目标位于 `input`、`textarea`、`select` 或 `contenteditable` 区域时，不拦截快捷键。
- 原因：避免插件面板或页面其他输入区域无法正常输入空格/方向键。
- 备选方案：无条件拦截；用户体验风险过高。

## Risks / Trade-offs

- [站点原生播放器也监听相同按键] → 通过仅在受管视频全屏上下文内处理，并在命中快捷键时调用 `preventDefault()` / `stopPropagation()`，降低双重响应风险。
- [跨 iframe/fullscreen 场景行为差异] → 以“拥有目标视频元素的内容脚本实例”作为监听主体，并围绕现有全屏判断做回归验证。
- [跳转后中文配音短暂重同步] → 复用现有 `seeked` 与漂移校正逻辑，避免新增独立同步分支。

## Migration Plan

1. 在内容脚本初始化阶段注册和清理键盘监听。
2. 为全屏状态、可编辑目标和按键映射增加判定函数。
3. 将快捷键操作统一落到 `video` 播放/暂停与 5 秒跳转上。
4. 在普通播放、全屏播放、启用中文配音和输入框聚焦场景下验证行为。
5. 如需回滚，只需移除新增键盘监听与辅助判定逻辑，不涉及存储或 Provider 协议迁移。
