## Context

当前中文配音在页面里通过 `new Audio(remoteUrl)` 直接播放 Provider 返回的音频文件。这个模式在同源或宽松安全策略下可行，但在 `https` 课程页面中访问 `http` Provider 音频资源时，浏览器可能因为 mixed content、安全策略或媒体源支持限制而拒绝播放。已经落地的分段输出能力也因此无法稳定发挥作用。

本次变更聚焦于“播放链路代理化”而不是“Provider 协议重构”。目标是在保持现有 Provider 输出结构不变的前提下，让扩展后台负责抓取音频二进制资源，再由内容脚本使用 `blob:` 或等价本地引用播放，并通过小窗口缓存降低内存风险。

## Goals / Non-Goals

**Goals:**
- 让中文配音播放不再依赖页面直接访问远端 Provider 音频 URL。
- 让分段播放与完整播放都能复用同一套 blob 代理播放链路。
- 使用小窗口缓存限制内存占用，并支持在切段/跳转后释放旧资源。
- 保持现有 Provider HTTP 接口与分段结果结构不变。

**Non-Goals:**
- 不修改 Python Provider 的协议或文件输出结构。
- 不实现整条视频音轨的全量预下载或离线缓存。
- 不在本次变更中引入 Service Worker 持久化缓存、IndexedDB 媒体存储或更复杂的媒体管道。

## Decisions

### 决策 1：由扩展后台抓取音频二进制，再转换为 `blob:` URL 供页面播放
内容脚本不再将远端 Provider URL 直接赋给 `Audio.src`，而是通过 `chrome.runtime.sendMessage` 请求后台抓取音频字节，并返回本地可播放引用。

**原因：**
- 避开页面上下文对远端 `http` 媒体源的安全限制。
- 保持 Provider 端无需感知页面安全上下文。
- 与当前已有的“后台代理文本 / JSON 请求”模式一致，便于集成。

**替代方案：**
- 让 Provider 升级为 `https`：可作为部署优化，但不能覆盖所有自定义 Provider 场景。
- 页面直接 `fetch` 二进制：仍可能受页面安全策略和跨域限制影响。

### 决策 2：采用“小窗口缓存”而不是全量缓存所有分段
插件仅保留当前播放段、下一段以及有限数量的邻近段；当播放位置变化或用户 seek 时，及时释放超出窗口范围的 blob 资源。

**原因：**
- 控制内存占用，不随视频总时长线性增长。
- 足以满足顺序播放和基础预取体验。

### 决策 3：缓存键以远端媒体 URL 为主，内容脚本负责窗口管理
后台代理可以按远端 URL 复用已生成的本地媒体引用；内容脚本根据当前分段索引决定哪些资源应保留、哪些资源应释放。

**原因：**
- 后台专注资源抓取与引用创建。
- 内容脚本最了解当前播放上下文，更适合决定缓存窗口。

### 决策 4：跳转后以“清空旧窗口并围绕目标重建”为默认策略
用户跳转到远离当前窗口的位置时，不尝试保留全部旧分段缓存，而是直接释放旧窗口并重新请求目标位置附近的分段。

**原因：**
- 降低状态复杂度。
- 避免缓存策略逐渐退化为全量保留。

## Risks / Trade-offs

- [频繁创建与释放 blob URL] → 用有限窗口和 URL 复用策略降低抖动，只在切段或 seek 时调整。
- [后台抓取增加消息与拷贝成本] → 仅对当前段与少量预取段执行抓取，不做整条下载。
- [播放前需要等待后台返回 blob] → 对下一段提前预取，尽量把等待隐藏在当前段播放期间。
- [实现复杂度增加] → 复用当前分段播放模型，只替换音频资源获取层，不同时重写更多状态机。

## Migration Plan

1. 为扩展后台增加二进制媒体抓取与 blob 引用返回能力。
2. 在内容脚本中加入按 URL 请求/复用/释放 blob 媒体引用的辅助层。
3. 将当前整段播放和分段播放统一切换到代理媒体源。
4. 增加窗口缓存与 seek 后窗口重建逻辑。
5. 验证 mixed content 场景下的实际播放效果与内存释放行为。

## Open Questions

- blob 引用应该全部在内容脚本侧创建，还是由后台返回可直接复用的引用标识？
- 小窗口默认保留 2 段、3 段还是按秒数控制更合适？
- 是否需要在调试面板里暴露当前缓存窗口状态，方便联调？
