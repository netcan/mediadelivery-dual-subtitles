## Context

当前插件已经通过 `localStorage` 保存字幕、Provider 和配音相关设置，也已经在 `loadedmetadata`、`seeked`、`timeupdate`、`play`、`pause` 等事件上维护字幕与中文配音同步。因此，“记住播放点”最自然的接入方式是扩展现有本地设置结构，并复用现有视频事件链路，而不是新增独立存储系统。

本次目标很明确：按视频记录 `currentTime`，刷新后恢复到上次位置，但不自动播放。这样既能保留观看进度，又不会撞到浏览器自动播放限制，也不会在恢复时引入额外的配音播放副作用。

## Goals / Non-Goals

**Goals:**
- 为当前视频记录上次播放位置，并在刷新后恢复。
- 使用按视频区分的 key，避免不同视频互相覆盖进度。
- 控制保存频率，避免在 `timeupdate` 上过度写入 `localStorage`。
- 恢复播放点后保持暂停状态，由用户手动继续播放。
- 让恢复行为与现有字幕/中文配音同步链路兼容。

**Non-Goals:**
- 不恢复“是否正在播放”的状态。
- 不实现跨设备同步或服务端存储。
- 不在本次引入播放历史列表、最近观看记录 UI 或手动管理面板。

## Decisions

### 1. 继续复用现有 `localStorage` 设置容器
- 决策：将播放点记录合并到现有设置对象中，而不是新建独立存储 key。
- 原因：当前插件已经有统一的读写入口，扩展该结构改动最小，也更便于后续增加其他本地状态。
- 备选方案：单独使用新的 `localStorage` key；可行但会分散状态管理。

### 2. 用“视频唯一标识”做 resume key
- 决策：基于 embed 页中的稳定标识（如路径中的视频 id/页面 URL/当前视频源）构造每个视频的 resume key。
- 原因：只有按视频分组，刷新后才能恢复到正确的播放点，且不同课程不会互相污染。
- 备选方案：只保存单个全局播放点；会导致不同视频覆盖彼此进度。

### 3. 在 `loadedmetadata` 后单次恢复播放点
- 决策：等视频元数据可用后再设置 `currentTime`，并确保每次初始化只恢复一次。
- 原因：需要先知道 duration 才能判断记录是否有效，也能避免反复把用户拖回旧位置。
- 备选方案：更早恢复；在 duration 未知时容易设置失败或恢复到错误位置。

### 4. 节流保存 + 关键时机强制保存
- 决策：在 `timeupdate` 上节流保存，并在 `pause`、页面隐藏或卸载前补充保存。
- 原因：兼顾性能和准确性，避免每次时间更新都写 `localStorage`。
- 备选方案：只在 `pause` 时保存；播放中直接刷新会更容易丢失最近进度。

### 5. 接近结尾的记录不自动恢复
- 决策：当记录位置距离视频结束只剩很短时间时，不自动恢复该位置。
- 原因：避免用户刷新后反复落在结尾附近，影响重播体验。
- 备选方案：无条件恢复；体验通常较差。

## Risks / Trade-offs

- [视频唯一标识不稳定] → 优先选择稳定路径/视频源组合，并提供兜底 key 生成方式。
- [保存过于频繁影响性能] → 使用节流与关键节点保存组合，而不是每次 `timeupdate` 都写入。
- [恢复时影响中文配音同步] → 让恢复通过 `video.currentTime` 触发既有同步链路，而不是单独操作外挂音频。
- [旧记录长期积累] → 首期可以接受；后续如有需要再加清理策略。

## Migration Plan

1. 扩展设置结构，增加按视频保存播放点的字段。
2. 增加 resume key 生成与记录读写辅助函数。
3. 在初始化后单次恢复播放点，并添加节流保存逻辑。
4. 在暂停和页面离开时补充保存进度。
5. 验证普通播放、刷新恢复、接近结尾和中文配音场景。
